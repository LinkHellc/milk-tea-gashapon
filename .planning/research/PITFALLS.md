# Pitfalls Research

**Domain:** 微信小程序奶茶扭蛋机工具
**Researched:** 2026/04/23
**Confidence:** MEDIUM

---

## Critical Pitfalls

### Pitfall 1: 概率算法不透明引发用户质疑

**What goes wrong:**
用户抽到"不想要的结果"后质疑概率被操控，在社区/评论区引发负面舆论。

**Why it happens:**
扭蛋机核心体验依赖随机性，但用户天然不信任"看不见的算法"。当结果不符合预期时，"黑箱概率"成为情绪出口。

**How to avoid:**
1. 公示各品牌的基础概率（如：隐藏款 5%，常规款 95%均分）
2. 提供"保底机制"（如：连续10次未抽到隐藏款，第11次必出）
3. 在结果页展示"今日抽取运气值"（可视化的随机种子展示，不透露具体算法）

**Warning signs:**
- 社区出现"小程序概率骗人"类反馈
- 用户评论出现"我再也不玩了"

**Phase to address:**
Phase 1 - 基础扭蛋体验（概率算法必须首期就设计好，不能后期添加）

### Pitfall 2: 数据持久化存储陷阱

**What goes wrong:**
用户刚攒了100积分，结果小程序缓存被清理，积分清零；或者添加的自定义奶茶款式全部丢失。

**Why it happens:**
微信小程序本地存储有10MB限制，且用户可在设置里手动清除。云开发存储也有容量和调用限制。积分/成就/用户自定义数据需要可靠存储方案。

**How to avoid:**
1. 核心用户数据（积分、成就、自定义奶茶）必须落云数据库，不只存本地
2. 设置存储预警（单用户数据超过500条时提示清理）
3. 本地只做缓存，云端才是source of truth
4. 自定义奶茶数据提供导出/备份机制

**Warning signs:**
- 用户反馈"我之前攒的分没了"
- 控制台出现存储配额警告

**Phase to address:**
Phase 2 - 用户系统与数据持久化

---

### Pitfall 3: 分享图生成失败切断传播链路

**What goes wrong:**
分享图生成是核心传播链，但Canvas绘图生成图片经常失败（白屏、QR码不识别、图像模糊），导致用户无法分享。

**Why it happens:**
1. wx.canvasToTempFilePath API 在部分机型上有兼容性问题
2. 小程序码生成依赖服务端，没有服务端则无法生成带小程序码的分享图
3. 跨域Canvas操作在真机上失败但开发者工具正常

**How to avoid:**
1. 不要依赖纯前端Canvas生成 - 使用微信官方提供的"标准分享图"接口
2. 或者：后端生成分享图，前端只做展示（推荐方案）
3. 必须真机测试所有机型（华为、小米、OPPO等国内机型碎片化严重）
4. 提供"复制链接"作为降级方案

**Warning signs:**
- 分享图在开发者工具正常，真机白屏
- 特定机型（尤其是华为）canvas API 返回空

**Phase to address:**
Phase 2 - 分享功能（首期必须完成且经过真机测试）

### Pitfall 4: 品牌商标侵权风险

**What goes wrong:**
使用"一点点"、"喜茶"、"茶百道"等品牌logo/名字作为扭蛋机选项，收到了品牌方的侵权投诉导致小程序被下架。

**Why it happens:**
微信平台对侵权投诉响应迅速，且品牌方（尤其是喜茶、霸王茶姬等）有专门的小程序维权团队。使用真实品牌名为核心功能会触发知识产权风险。

**How to avoid:**
1. v1阶段使用通用奶茶类型名（如"港式奶茶"、"水果茶"）而非具体品牌名
2. 如需使用品牌，使用品牌授权或选择无版权争议的开放式品牌列表
3. 或者：使用"代号"系统（如"品牌A"、"品牌B"），用户自己对应
4. 在小程序简介/说明中明确"本程序内品牌logo仅供展示，不代表品牌官方合作"

**Warning signs:**
- 收到微信官方的侵权投诉通知
- 品牌方直接联系要求下架

**Phase to address:**
Phase 1 - 必须在首期就确定是否使用真实品牌，风险必须前置评估

### Pitfall 5: 动画性能崩溃毁掉核心体验

**What goes wrong:**
扭蛋动画卡顿、掉帧、动画过程中手机发烫，核心体验变成"打开小程序等5秒动画"。

**Why it happens:**
1. 微信小程序WXML/CSS动画性能远不如Web，在老旧机型上更明显
2. 开发者常用setTimeout/requestAnimationFrame做动画，在真机上与原生渲染冲突
3. 同时运行多个动画（如胶囊弹射+奶茶名称打字机效果）导致CPU过载

**How to avoid:**
1. 使用微信提供的wx.createAnimation而非CSS动画
2. 动画控制在500ms以内，复杂动画分帧
3. 老旧机型（3年前的手机）提供"简化动画"降级方案
4. 关键动画必须有"跳过"按钮

**Warning signs:**
- 开发者工具流畅，真机测试卡顿
- 用户反馈"每次打开都卡"
- iPhone 6/7等老机型上帧率低于30fps

**Phase to address:**
Phase 1 - 扭蛋动画必须在首期完成并做真机性能测试

---

### Pitfall 6: 广告变现节奏失衡

**What goes wrong:**
广告加载太慢导致用户体验断层，或者广告太激进被用户举报，还可能广告主内容与奶茶主题产生反差（比如抽到奶茶结果底下是个整容广告）。

**Why it happens:**
1. 激励视频广告wx.createRewardedVideoAd有加载延迟，用户点击到看到广告可能有3-5秒黑屏
2. 工具型小程序"即用即走"特性要求广告必须轻量，但变现需求又要求广告存在
3. Banner广告位置不当会误触，导致分享图被截断

**How to avoid:**
1. 广告只在"结果页"底部出现，不在抽取流程中出现
2. 激励视频广告入口放在"下次扭蛋前"，用户主动点击，不是强制观看
3. 设置广告超时兜底（3秒未加载完成则隐藏广告位）
4. 选择广告主时优先食品/饮品/生活方式类，避免反差过大的广告

**Warning signs:**
- 用户反馈"小程序里全是广告"
- 激励视频广告加载失败率超过20%

**Phase to address:**
Phase 3 - 广告系统（必须在用户习惯养成后再接入，避免首期就赶跑用户）

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| 本地存储存积分，暂不上云 | 快速开发，早期数据少无感 | 用户积累数据后清空，信任崩塌 | 仅MVP验证阶段，且最多100用户规模 |
| 简单Math.random()做概率 | 3行代码实现随机 | 无法验证公平性，无法做保底机制 | 永远不推荐 |
| Canvas截图生成分享图 | 前端搞定，无需后端 | 机型兼容性差，维护成本高 | 仅MVP快速验证 |
| 所有动画用setTimeout链式调用 | 代码简单，写起来快 | 老旧机型卡死，难以调试 | 仅原型验证 |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| 微信小程序码生成 | 期望在前端生成带参数的小程序码 | 必须通过 wx.createQRCode 小程序码接口，服务端生成 |
| 微信支付（积分充值） | 直接调用wx.requestPayment | 需要商户资质，且虚拟商品支付有政策限制，v1不建议做付费充值 |
| 广告联盟接入 | 直接在页面放广告组件 | 需要先申请微信广告资质，且内容类小程序有更严格审核 |
| 分享到朋友圈 | 期望用户能直接分享到朋友圈 | 朋友圈分享仅支持小程序卡片，不支持自定义分享图 |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| 一次性加载所有品牌logo | 首屏加载超过3秒，用户流失 | 图片压缩+按需加载，首屏只加载用户可见的3-4个品牌 | 用户网络慢于4G时 |
| 每个品牌扭蛋机页面独立加载奶茶数据 | 切换品牌时明显卡顿 | 数据预加载+本地缓存，数据切换秒完成 | 9个品牌全部加载时 |
| 动画使用CPU渲染 | 动画时手机发烫，耗电快 | 使用微信动画API而非纯CSS动画 | 老旧机型（3年以上机龄） |
| 云数据库高频查询 | 云函数调用慢，计费高 | 本地缓存+增量更新，而非每次拉取全量数据 | 100并发以上用户 |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| 前端存储用户积分余额 | 用户可通过JS修改本地积分 | 积分计算必须在云函数/云数据库完成，前端仅展示 |
| 概率算法纯前端实现 | 用户可抓包修改随机种子 | 算法核心逻辑必须在云端，前端只触发和展示结果 |
| 隐藏款概率可配置 | 配置错误导致概率失控，或被恶意配置 | 云端常量+配置需双重校验，不在前端可写 |
| 分享图含用户敏感信息 | 未经授权展示用户头像/昵称违反隐私政策 | 分享图只含奶茶信息和程序码，不含个人信息 |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| 扭蛋动画不可跳过 | 用户每次都要看3-5秒动画，重复使用烦躁 | 提供"再来一次"快速通道，动画1秒内完成 |
| 积分看不到来源和用途 | 用户不知道为什么有积分，积分有什么用 | 每获得积分时toast提示"获得+X积分，攒满100可兑换XXX" |
| 抽到重复奶茶无反馈 | 用户觉得"我是不是抽到过这个" | 抽到已收集奶茶时展示"你之前抽到过，+1积分作为补偿" |
| 品牌列表无搜索/筛选 | 用户在9个品牌里找半天 | 默认按"最近使用"排序，提供简短搜索 |
| 结果页无"下一步"引导 | 用户分享完就关掉小程序，没有留存 | 结果页提供"再扭一次"或"查看积分商城"入口 |

## "Looks Done But Isn't" Checklist

- [ ] **分享图:** 在开发者工具正常 - 真机上白屏了吗？华为/OPPO/小米各测过吗？
- [ ] **概率算法:** 纯前端Math.random() - 云端算法实现了吗？
- [ ] **积分系统:** 本地缓存 - 云数据库同步了吗？用户换手机积分还在吗？
- [ ] **动画性能:** 开发者工具流畅 - 真机（尤其是老旧机型）帧率如何？
- [ ] **广告加载:** 激励视频广告 - 3秒内能加载出来吗？超时兜底做了吗？
- [ ] **品牌商标:** 使用了真实品牌名 - 侵权风险评估做了吗？
- [ ] **隐私政策:** 广告变现 - 隐私政策文档准备好了吗？

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| 概率质疑风波 | HIGH | 紧急更新公示概率算法，发布"保底机制"，联系平台删帖控评 |
| 数据丢失 | HIGH | 从云端备份恢复（如果有云存储），否则用户补偿（手动补发积分） |
| 分享图生成失败 | MEDIUM | 降级为"复制链接"，快速修复Canvas兼容性问题，审核后紧急发布 |
| 商标侵权投诉 | CRITICAL | 立即下架相关品牌，改用通用名，申请品牌授权或等待投诉处理完成 |
| 广告体验差被举报 | MEDIUM | 降低广告频次，移除激进广告位，提交申诉说明整改方案 |

## Pitfall-to-Phase Mapping

| Pitfall | Prevention Phase | Verification |
|---------|------------------|--------------|
| 概率算法不透明 | Phase 1 | 用户反馈无"概率骗人"投诉 |
| 数据持久化 | Phase 2 | 用户换手机后积分/数据仍存在 |
| 分享图生成 | Phase 2 | 华为/小米/OPPO真机测试通过 |
| 品牌商标风险 | Phase 1 | 获得品牌方书面授权或确认使用通用名 |
| 动画性能 | Phase 1 | 老旧机型（iPhone 7）动画帧率>30fps |
| 广告变现失衡 | Phase 3 | 用户留存率在广告上线后无明显下降 |
